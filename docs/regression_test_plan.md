# JOJIFrontier 現行基盤・回帰テスト計画

文書種別: **進捗記録**
仕様索引: [`README.md`](README.md)
この文書は試験範囲を管理する。期待値の定義元は各ゲーム仕様正本とする。

更新日: 2026-07-13

この文書は全体ロードマップPhase 0の完了条件を定める。ゲーム仕様の正本ではなく、既存機能を
今後の共通基盤変更から保護するテスト契約である。

## テスト方針

- ロジックテストはraylibを起動せず、固定Seedで実行する
- 1テストは1つの失敗理由を持つ
- ランダム性は確率ではなく、同じSeedの再現性と多数Seedの不変条件を分けて検証する
- UI文字列ではなく状態、ID、数値を検証する
- バグ修正時は再現テストを先に追加する
- スクリーンショット確認だけで戦闘ロジックの合否を決めない
- Test targetはBuild typeにかかわらず`NDEBUG`を解除し、定義された状態ならCompile errorにする
- `assert(functionWithSideEffect())`へSetupやActionを入れない。Actionを先に実行し、結果だけを検証する
- Test runnerが実行したTest数と失敗数をCI Logへ残す

### 過去の検証記録

2026-07-13以前のデフォルト`RelWithDebInfo`では`NDEBUG`により`assert()`が無効化され、
副作用を持つSetupやActionまで実行されないTestが存在した。この期間の「Test成功」は単独では
回帰保証として扱わない。Phase 1以降の基盤は`-UNDEBUG`修正後に再実行した結果を基準とする。
既存機能を変更する時は、対象Testが実際に失敗し得ることを一度確認する。

## A 戦闘入力状態

1. Unit選択から移動、攻撃、待機まで正常遷移
2. 移動前キャンセルでUnit選択へ戻る
3. 移動後の戻るで元位置と未行動状態を復元
4. 攻撃確認キャンセルで行動選択へ戻る
5. 行動確定後は同じUnitを再行動できない
6. 全生存味方行動後にEnemy Phaseへ移る
7. Enemy Phase終了後に生存味方の行動済み状態を解除
8. Victory/Defeat後に戦闘入力を受け付けない

## B 移動と占有

1. 味方マスを通過可能
2. 味方マスへ停止不可
3. 敵マスを通過・停止不可
4. 通行不可地形を経路に使用しない
5. 地形移動コストを到達範囲へ反映
6. 移動後は移動範囲を非表示
7. 上下移動を含む最短距離が正しい
8. Zone of Controlが対応Unitだけに作用
9. 初期配置時だけ足元の通行不可地形を通常床へ変更

## C 攻撃と表示用範囲

1. 移動前は全到達マスからの攻撃可能範囲を生成
2. 移動後は現在地からの攻撃可能範囲だけに縮小
3. 味方Unitを攻撃対象にしない
4. 敵不在マスも攻撃可能範囲へ含む
5. 実際に攻撃可能な敵マスだけTargetableへ含む
6. 基本必中、茂み上だけ命中80%
7. 同じSeedと攻撃順で命中結果を再現
8. 最低ダメージ1
9. 物理はDEF、魔法はRESを参照

## D 地形と強制移動

1. 薬草地点で行動終了するとHP5回復して通常床へ変化
2. HP満タンでも薬草地点を消費
3. 浅瀬の移動コスト2
4. 茂みの回避補正を敵味方双方へ適用
5. Heavy Spearが直交方向へ1マス押し出す
6. 盤外、Unit、通行不可地形がある場合は押し出さない
7. 獣皮の柄巻きが最初のノックバックだけを無効化
8. 防護板の配置候補と通行阻害

## E Phase・状態異常・スキル

1. Phase開始時のスキル使用回数更新
2. Phase終了時の毒・炎上処理順
3. 毒でHP1未満にならない
4. 炎上で戦闘不能になり得る
5. 同名状態異常を重複せず再付与規則に従う
6. 戦闘終了時に全状態異常を解除
7. スキル使用条件、対象、使用回数、行動消費
8. 敵Phaseでも共通状態処理を使用

## F 遠征と画面遷移

1. 拠点から4人・6枠確定後だけ遠征開始
2. 探索結果を戦闘生成へ1回だけ適用
3. 戦闘勝利後だけキャンプへ進める
4. キャンプ続行でHP、戦闘不能、バッグ残数を引き継ぐ
5. 安全帰還でPendingをStorage/Registryへ確定
6. 敗北と手動撤退でPendingを全破棄
7. 拠点帰還で全員のHPと戦闘不能を回復
8. 帰還後に同じ勝利報酬を二重付与しない

## G 拠点、施設、装備

1. 素材不足時に一部素材だけ消費しない
2. 建設した施設は切替なしで常時利用できる
3. 建設、研究、所有の各状態を分離して保持
4. 未建設施設では新規研究・製作不可
5. 所有済み装備・スキル・特性・消耗品・探索道具を常に準備可能
6. 旧`builtNodeIds`を建設済み施設へ移行し、旧稼働状態を読み捨てられる
7. 未解放レシピの武器を装備不可
8. 装備、特性、スキルをUnit単位で保持

## H セーブ

1. Schema 2 JSONラウンドトリップ
2. Schema 1の兵種装備をSchema 2のUnit装備へ移行
3. 通常セーブ破損時にバックアップ読込
4. 将来Schemaを拒否し、自動上書きしない
5. 拠点、倉庫、Discovery、施設、編成、装備、言語を復元
6. Exploration/Campチェックポイントを復元
7. 戦闘中終了時は直前チェックポイントへ戻る
8. 同じExpedition Seedで戦闘を再生成

## I ランダム生成

固定Seedテスト:

- 同じ地域ID、地点ID、戦闘番号、Seedから同じ24マスを生成
- 同じ敵編成と初期配置を生成
- リロードで再抽選しない

多数Seed不変条件テスト:

- 味方から敵側へ最低1本の通行可能経路
- 3行すべてを同じ列で封鎖しない
- 初期配置が重複しない
- Objectiveと特殊Objectへ到達可能
- 生成再試行回数が上限内

テスト用Seed一覧は固定し、失敗したSeedを削除せず回帰ケースへ追加する。

## J 日本語UIの静的検査

- 通常画面に内部ID、Seed、英語兵種名を表示しない
- 日本語文字列の欠落を`?`へ置換しない
- ボタン、ツールチップ、施設説明の文字切れを主要解像度で確認
- 同じメッセージを本文とボタンラベルで重複表示しない
- UI確認はデスクトップとWebの両方で行う

## Phase 0完了判定

次をすべて満たした時だけPhase 0完了とする。

1. Debug/Releaseビルド成功
2. CTest全成功
3. AddressSanitizerまたは同等のDebug検査で主要フローを完走
4. 固定Seedと多数Seedの生成テスト成功
5. Schema 1/2の保存Fixtureテスト成功
6. 日本語主要画面の目視確認
7. 既知の失敗を無効化・コメントアウトしていない

Web実ビルド、IndexedDB同期、GitHub Pages更新試験はPhase 12で扱い、Phase 0のブロッカーには
しない。ただしデスクトップで再現する保存・ロジック不具合はPhase 0で解消する。
